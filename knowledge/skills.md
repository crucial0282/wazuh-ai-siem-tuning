# Wazuh SIEM Detection Engineering Knowledge Base

## Objective

Analyze Wazuh alert summaries and generate accurate, safe custom detection rules.

Focus:
- Reduce alert noise
- Detect real threats
- Avoid suppressing important security events

---

## Rule Severity Guidelines

Level scale:

0 → Ignore  
1–3 → Low (system noise)  
4–6 → Low-medium (user errors, expected activity)  
7–9 → Medium (notable activity)  
10–11 → High (likely threat)  
12–13 → Critical (confirmed malicious behavior)  
14–15 → Severe (APT / major compromise)

Rules generated by the AI:
- Must be level 7 or higher
- Should use level 10+ only for strong threat indicators
- Should not assign level 15 unless the event represents an extremely severe and high-confidence threat

---

## Rule ID Policy

- Never use IDs below 100000 for generated custom rules
- Always generate IDs in the range:
  100000–109999
- IDs must be unique
- Never reuse previously suggested IDs
- Use only rule IDs provided by the rule ID reservation system

---

## Safe Detection Engineering Rules

Never blindly suppress or downgrade security events related to:

- Authentication failures
- Privilege escalation
- Brute-force attempts
- Suspicious network connections
- Malware execution
- Persistence activity
- Credential access
- Suspicious administrative activity

Instead:

- Narrow the rule using specific filters
- Add conditions based on relevant fields
- Filter known benign processes when appropriate
- Consider source IP, destination IP, username, process, agent, or other available context
- Reduce severity only when activity is clearly understood and considered benign

Always prefer precise detection logic over broad suppression.

---

## Alert Frequency Guidelines

A frequently triggered rule is not automatically noise.

Before recommending tuning:

1. Examine the rule description
2. Examine the rule severity
3. Check the affected agents
4. Check source IP patterns when available
5. Check associated MITRE ATT&CK information
6. Compare current activity with historical patterns
7. Determine whether the activity represents expected behavior or suspicious repetition

High frequency alone must never be used as the only reason to suppress a rule.

---

## Common SIEM Patterns

Examples of patterns that may appear in Wazuh alerts:

- PAM session events may occur frequently during normal Linux activity
- Process creation events may generate high volumes from legitimate applications
- Repeated SSH authentication failures may indicate brute-force activity
- Repeated authentication failures from the same source IP may indicate password guessing
- Privilege escalation events should be reviewed carefully
- Repeated alerts from a single agent may indicate either abnormal behavior or a noisy configuration
- Sudden increases in rule frequency may indicate a change in system behavior or a potential attack

Always use available context before recommending rule changes.

---

## Wazuh Rule Format

Generated rules must use valid Wazuh XML syntax.

Basic example:

<rule id="CUSTOM_RULE_ID" level="LEVEL">
  <if_sid>PARENT_RULE_ID</if_sid>
  <field name="FIELD_NAME">VALUE</field>
  <description>RULE_DESCRIPTION</description>
</rule>

---

## Rule Writing Requirements

Every generated rule must:

- Be a single valid `<rule>` element
- Include a unique rule ID
- Include a valid Wazuh severity level
- Include a meaningful `<description>` element
- Use valid XML syntax
- Be concise and specific
- Avoid overly broad matching conditions
- Be based on evidence available in the alert summary
- Avoid inventing fields that are not supported by the available alert data

When extending an existing Wazuh rule, use the appropriate parent rule relationship when applicable.

Example:

<rule id="100001" level="10">
  <if_sid>5503</if_sid>
  <field name="srcip">EXAMPLE_IP</field>
  <description>Repeated authentication failure activity from monitored source</description>
</rule>

---

## Detection Strategy

When analyzing alert summaries:

1. Identify frequently triggered rules
2. Identify high-severity alerts
3. Identify repeated activity from source IP addresses
4. Identify agents producing unusual alert volumes
5. Review MITRE ATT&CK mappings when available
6. Compare current activity with historical patterns
7. Identify new patterns that were not previously observed
8. Determine whether an existing rule already provides sufficient detection
9. Suggest a new rule only when it provides meaningful additional detection value

Do not generate rules simply because an alert occurred frequently.

---

## Noise Reduction Strategy

When addressing alert noise:

Prefer:

- Narrowing detection conditions
- Filtering known benign behavior
- Adjusting severity when justified
- Adding context-specific child rules

Avoid:

- Disabling security rules entirely
- Suppressing authentication failures without investigation
- Suppressing privilege escalation events
- Suppressing malware-related detections
- Creating broad level-0 suppression rules without strong justification

---

## MITRE ATT&CK Guidance

When MITRE ATT&CK information is available:

- Use it as supporting context
- Do not invent MITRE technique IDs
- Ensure the technique is relevant to the detected behavior
- Do not classify normal administrative activity as malicious solely because it maps to a MITRE technique

MITRE mappings should improve detection context, not replace alert analysis.

---

## AI Rule Generation Behavior

The AI acts as an assistant to a detection engineer.

The AI should:

- Analyze alert patterns
- Identify potential detection gaps
- Suggest focused detection rules
- Explain why each rule is useful
- Mention potential false positives
- Generate syntactically valid Wazuh XML

The AI should not:

- Automatically assume high-frequency alerts are malicious
- Automatically assume low-frequency alerts are safe
- Invent unsupported alert fields
- Disable important security detections
- Claim that generated rules are safe for production without testing

---

## Output Expectations

Generated rules must be:

- Actionable
- Focused
- Syntactically valid
- Based on observed alert patterns
- Compatible with Wazuh rule structure
- Suitable for analyst review

All AI-generated rules must be reviewed and tested before deployment to a production Wazuh environment.
